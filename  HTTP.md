## HTTP
HTTP는 클라이언트와 서버가 데이터를 주고받기 위한 통신 규칙이다.  
웹에서 요청과 응답 방식으로 동작한다.
(HTTP와 HTTPS의 차이: HTTP는 암호화하지 않고 데이터를 그대로 전송, 중간에서 볼 수 있다. 반면 HTTPS는 데이터를 암호화하여 안전하다.)

### 특징
- Stateless: 각 요청은 서로 독립적이며 이전 상태를 기억하지 않는다.
- 텍스트 기반 프로토콜이라 구조를 사람이 읽을 수 있다.
- 주로 TCP 기반 위에서 동작한다.

## HTTP method
HTTP Method는 클라이언트가 서버에 어떤 동작을 요청하는지 나타낸다.

### 특징
- 같은 URL이라도 method에 따라 수행되는 동작이 달라진다.
- 자원의 상태를 어떻게 변경할지 정의하는 역할을 한다.

### 주요 메서드
- GET: 데이터 조회
- POST: 데이터 생성
- PUT: 데이터 전체 수정
- PATCH: 데이터 일부 수정
- DELETE: 데이터 삭제


## API
API는 서로 다른 프로그램이 통신할 수 있도록 해주는 인터페이스이다.  
보통 HTTP를 기반으로 요청과 응답 형태로 데이터를 주고받는다.
### 장점
- 표준화 및 연결성: 서로 다른 플랫폼이나 언어 간 호환성을 제공하여 시스템 간 연동을 원활하게 한다.
- 보안 및 데이터 보호: 내부 시스템 전체를 노출하지 않고, 정의된 API를 통해 필요한 데이터만 제한적으로 공유하여 보안을 강화한다.
- 구현 방식 은닉: 내부 시스템이 어떻게 작동하는지 알지 못해도(블랙박스), 명세에 따라 요청하면 원하는 결과값을 받을 수 있다.
- 자동화 및 효율성: 수작업을 줄여 대용량 데이터 처리나 주식 주문 등 빠르고 정확한 작업을 자동화한다.
- 유연성 및 확장성: 기존 툴을 관리하거나 새로운 서비스를 설계할 때 유연성을 제공하여 시스템을 확장하기 쉽다.

 ### Interface(인터페이스)
서로 다른 두 시스템, 장치, 또는 사람과 사물 사이에서 정보나 신호를 주고받는 접점이나 경계면을 의미한다.

### 특징
- 요청과 응답 구조로 동작한다.
- JSON 형식을 많이 사용한다.
- 프론트엔드와 백엔드를 연결하는 역할을 한다.

## MVC
소프트웨어 디자인 패턴 중 하나로, 사용자 인터페이스(UI)와 비즈니스 로직을 분리하여 애플리케이션의 유지보수성, 재사용성, 확장성을 높이는 개발 방법론이다.

### Model
데이터와 비즈니스 로직을 처리한다. 데이터베이스와의 상호작용을 담당하며, 상태 변경이 발생하면 뷰나 컨트롤러에 알린다.

### View
사용자에게 보여지는 UI를 담당한다. 모델이 처리한 데이터를 화면에 표시하며, 사용자의 입력(이벤트)을 컨트롤러로 전달한다.

### Controller
모델과 뷰 사이의 중재자이다. 사용자의 입력을 받아 모델의 로직을 호출하고, 모델의 결과물을 뷰에 전달하여 화면을 갱신한다
